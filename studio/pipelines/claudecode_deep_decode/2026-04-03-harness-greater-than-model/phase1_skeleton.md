# Phase 1: 结构解剖

## 1. 核心论点（一句话）

Claude Code 的竞争力不在 Claude 模型本身，而在围绕模型构建的 51 万行工程基础设施（Harness）——通过前缀沉重架构实现 92% 缓存复用、五步预处理流水线、三层上下文压缩，把"能力强但不可预测"的 LLM 变成"稳定可靠能交付"的 Agent。

## 2. 隐含假设

- **模型能力已"够用"**：作者假设当前 LLM 的推理能力足够完成大部分编程任务，瓶颈已从"模型不行"转移到"模型周围的系统不行"
- **工程复杂度是正当投入**：51 万行代码不是过度工程化，而是必要的安全、可靠性成本
- **缓存复用可以被系统性设计**：92% 不是偶然结果，是架构级别的刻意优化
- **Harness 可以跨模型迁移**：隐含假设这套系统换个同级别模型也能运作

## 3. 关键证据

1. **冰山比例**：51.2 万行代码中，直接与 LLM 交互的核心部分（query.ts + QueryEngine.ts + claude.ts）仅约 6,400 行。剩余 50 万行全是让 Agent 安全、高效运行的工程基础设施
2. **92% 缓存复用率**（LMCache 实测）：92 次 LLM 调用处理约 200 万 token，通过子 Agent 隔离 + 并行执行 + 前缀稳定性设计，实现 81% 成本节省
3. **五步预处理流水线**（源码实证）：Snip → 微压缩 → 上下文折叠 → 自动压缩 → 组装请求，每步有精心设计的依赖关系
4. **grep 而非向量数据库**：代码搜索用 ripgrep，不用 Embedding——最简单最可靠的工具做最关键的事

## 4. 盲区

- **自家 API 的独占优势**：prompt cache 是 Anthropic API 独有特性，92% 复用率在第三方部署中无法复现，这让"Harness 可迁移"的论点打折扣
- **维护成本的沉默代价**：51 万行 TypeScript 的长期维护、技术债务、团队协调成本被忽略
- **模型-Harness 共进化耦合**：Harness 的设计高度依赖 Claude 模型的特定行为特征，换模型后可能需要大幅调整
- **开源替代品的追赶速度**：泄露后 shareAI-lab/learn-claude-code 获 47.6k stars，"从零构建 agent harness"正在被快速平民化

## 5. 读者画像

- **目标读者**：AI 产品经理、技术决策者、Agent 架构师——关心"AI 产品竞争力到底在哪"
- **对我们的意义**：我们重度使用 Claude Code，理解 Harness 架构有助于：(1) 更好地配置 CLAUDE.md/Skills/Hooks 以最大化 Harness 效能；(2) 判断 Agent 产品的真正技术壁垒在哪；(3) 评估自建 Agent 系统时的工程投入量级
