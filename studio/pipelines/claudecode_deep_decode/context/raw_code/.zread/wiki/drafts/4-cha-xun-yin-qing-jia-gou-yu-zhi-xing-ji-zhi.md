查询引擎是 Claude Code 的核心执行单元，负责管理对话生命周期、处理用户输入、协调 API 调用、执行工具、维护会话状态，并实现高级特性如上下文压缩、Token 预算控制和错误恢复。本文档深入解析查询引擎的架构设计、核心组件交互机制以及关键执行流程。

## 架构概览与核心组件

查询引擎采用分层架构设计，顶层是 **QueryEngine 类**，作为会话管理器负责整个对话生命周期的状态管理；中间层是 **query 函数**，作为核心执行循环实现消息处理、API 调用和工具编排；底层是 **依赖注入模块**，提供 API 调用、压缩算法等核心能力的抽象接口。这种分层设计使得测试可以注入 mock 依赖，生产环境使用真实实现，同时保持核心逻辑的纯净性。

**QueryEngine 类**是每个对话的独立实例，持有完整的会话状态包括消息历史、权限拒绝记录、使用量统计、文件读取缓存和技能发现状态。其核心方法 `submitMessage` 接收用户输入（字符串或内容块数组），通过生成器异步产出 SDK 消息流，最终返回执行结果。该类封装了会话持久化、权限包装、系统提示构建、孤立权限处理等横切关注点，使上层调用者无需关注底层细节。

**query 函数**是查询引擎的核心执行循环，采用生成器函数设计实现流式响应处理。该函数维护一个可变状态对象，在每次迭代中处理消息压缩、API 调用、工具执行和停止钩子，通过 `yield` 将中间结果传递给调用者，通过 `continue` 在满足特定条件时重试迭代，最终返回终止状态。这种状态机设计使得复杂的查询流程可以被清晰建模，同时支持错误恢复和重试机制。

**依赖注入模式**通过 `QueryDeps` 类型定义核心依赖接口，包括 API 调用函数、压缩算法、UUID 生成器等。生产环境使用 `productionDeps()` 提供真实实现，测试环境注入 mock 函数以隔离外部依赖。这种模式避免了模块级别的 spy 调用，使测试代码更加简洁可控，同时保持了类型安全性和签名自动同步。

## 执行流程与状态管理

查询引擎的执行流程遵循明确的阶段划分，从用户输入处理开始，经过消息预处理、上下文压缩、API 调用、工具执行，最终到结果产出和清理。每个阶段都有清晰的输入输出契约，通过生成器实现非阻塞的流式处理。

**用户输入处理阶段**首先处理孤立权限（orphaned permission），这是指在之前会话中断时未完成的权限请求。然后通过 `processUserInput` 函数解析用户输入，处理斜杠命令、文件附件、内存引用等特殊语法，生成规范化的消息对象。该阶段还会触发技能发现和插件加载，为后续的 API 调用准备完整的工具集。处理完成后，消息被追加到 `mutableMessages` 数组，并持久化到会话存储，确保即使进程在 API 响应前被终止，对话也可以从用户消息点恢复。

**消息预处理阶段**执行一系列优化和压缩操作，包括工具结果预算应用、历史记录裁剪、微压缩和上下文折叠。工具结果预算通过 `applyToolResultBudget` 限制单个工具结果的字符数，防止过大的工具输出消耗过多 token。历史记录裁剪通过 `snipCompactIfNeeded` 移除旧消息，基于启发式规则识别可以安全删除的对话片段。微压缩通过 `microcompactMessages` 对工具调用序列进行紧凑表示，使用摘要替换详细的工具输出。上下文折叠通过 `applyCollapsesIfNeeded` 将相关消息分组折叠，在保留语义的同时减少 token 消耗。

**API 调用阶段**通过 `queryModelWithStreaming` 向 Anthropic API 发送请求，使用流式响应处理避免长时间阻塞。该阶段处理模型选择、提示缓存、思考模式、快速模式等高级特性，同时监控响应中的错误信号如 `prompt_too_long` 和 `max_output_tokens`，并在必要时触发模型降级或重试。流式响应中的每个内容块被即时处理，助手消息被追加到 `assistantMessages` 数组，工具调用块被收集到 `toolUseBlocks` 数组，为后续工具执行做准备。

**工具执行阶段**根据配置选择流式执行或批量执行模式。流式执行模式通过 `StreamingToolExecutor` 在助手消息流式到达时立即开始工具执行，与 API 响应并行运行，显著降低端到端延迟。批量执行模式在 API 响应完全接收后一次性执行所有工具调用，逻辑更简单但延迟更高。工具执行涉及权限检查、参数验证、沙箱隔离和结果规范化，生成的用户消息（工具结果）被追加到 `toolResults` 数组，并通过 `yield` 传递给上层调用者。

**错误恢复阶段**处理多种错误场景，包括提示过长、输出 token 超限、媒体大小超限等。对于提示过长错误，引擎首先尝试排空已暂存的上下文折叠，然后尝试响应式压缩，通过智能摘要减少 token 消耗后重试。对于输出 token 超限错误，引擎首先升级最大输出 token 限制并重试，如果仍然超限则注入恢复消息引导模型分步完成任务。媒体大小错误通过响应式压缩的剥离重试机制处理，移除大型附件后重新发送请求。

**状态管理机制**通过可变状态对象在查询循环迭代间传递信息。状态对象包含当前消息列表、工具使用上下文、自动压缩跟踪、恢复计数、工具使用摘要等字段。每次迭代开始时解构状态对象，迭代结束时通过 `continue` 语句和新的状态对象重新开始循环，或通过 `return` 语句终止循环。这种模式避免了深层嵌套和复杂的状态传递，使代码逻辑更加线性清晰。

## 核心机制详解

### Token 预算控制

Token 预算控制机制通过 `BudgetTracker` 和 `checkTokenBudget` 函数实现，用于在长会话中自动管理 token 消耗，防止超出预算限制。预算跟踪器记录连续计数、上次增量、上次全局轮次 token 和开始时间戳，每次 API 响应后检查是否需要继续执行。

预算检查逻辑首先判断是否适用预算控制：如果存在 agentId（子代理）或预算为 null/负值，直接返回停止决策。对于适用的情况，计算当前轮次 token 占预算的百分比和自上次检查以来的增量。如果增量持续低于阈值（低于 500 tokens）且连续计数超过 3 次，标记为边际效益递减，避免无意义的重复调用。

当完成度低于 90% 且未出现边际效益递减时，返回继续决策并生成继续消息，通过注入系统提示引导模型继续完成任务。当达到完成度或出现边际效益递减时，返回停止决策并生成完成事件，记录整个预算执行过程的统计信息。

这种机制特别适用于有 token 预算限制的自动化任务，通过智能判断何时继续、何时停止，在预算约束下最大化任务完成度。

Sources: [tokenBudget.ts](src/query/tokenBudget.ts#L1-L94)

### 停止钩子处理

停止钩子是在每个查询轮次结束时执行的回调函数，用于执行后处理任务如内存提取、提示建议、自动梦想、模板分类等。停止钩子通过 `handleStopHooks` 函数处理，该函数接收当前消息列表、助手消息、系统提示、用户上下文、工具使用上下文等参数，通过生成器产出钩子进度和结果消息。

停止钩子上下文首先保存缓存安全参数，使得后续的 `/btw` 命令或 SDK 控制请求可以访问当前会话状态。然后根据配置执行各种后台任务：模板作业分类器在每次轮次后更新作业状态，内存提取服务在后台提取会话中的记忆，自动梦想服务在后台生成后续任务建议，芝加哥 MCP 清理函数释放计算机使用锁和恢复界面可见性。

钩子执行通过 `executeStopHooks` 函数协调，该函数接收权限模式、中止信号、钩子激活标志等参数，通过生成器产出进度消息和结果。阻塞错误会导致钩子返回错误消息，阻止继续执行；非阻塞错误仅记录日志，不影响后续流程。钩子执行结果包括阻塞错误列表和是否阻止继续执行的标志，这些信息被用于决定是否触发下一轮查询迭代。

停止钩子机制使得扩展功能可以在不修改核心查询逻辑的情况下注入，保持了核心引擎的简洁性和可维护性，同时支持丰富的后处理能力。

Sources: [stopHooks.ts](src/query/stopHooks.ts#L1-L200)

### 配置快照与依赖注入

查询配置通过 `buildQueryConfig` 函数在查询入口处快照，包含会话 ID 和运行时门控标志。运行时门控包括流式工具执行、工具使用摘要发射、蚂蚁模式、快速模式等，这些标志来自环境变量或 Statsig 特性开关，在查询期间保持不变。

配置快照的目的是为未来的状态机重构做准备，通过将不可变配置与可变状态分离，使得纯函数式的事件处理器可以接收和状态元组，实现更好的可测试性和可推理。

依赖注入通过 `QueryDeps` 类型定义核心依赖接口，生产环境使用 `productionDeps` 提供真实实现，测试环境注入 mock 函数。当前定义的依赖包括 API 调用函数、微压缩函数、自动压缩函数和 UUID 生成器。这种模式避免了模块级别的 spy 调用，使测试代码可以精确控制外部依赖的行为，同时保持类型安全性。

## 消息处理与转换

### 消息规范化流程

查询引擎在多个阶段对消息进行规范化处理，确保消息格式符合 API 要求和内部约定。`normalizeMessagesForAPI` 函数将内部消息对象转换为 API 消息参数格式，处理工具结果配对、内容块转换和字段清理。`normalizeMessage` 函数在 QueryEngine 中将内部消息转换为 SDK 消息格式，添加会话 ID、时间戳和重放标志。

消息规范化处理多种消息类型：助手消息包含内容块列表和使用量统计，用户消息包含文本内容和可选的附件引用，系统消息包含子类型和元数据，进度消息包含工具执行状态，墓碑消息用于标记应删除的消息。每种消息类型都有特定的规范化逻辑，确保下游消费者接收到一致的格式。

消息持久化通过 `recordTranscript` 函数实现，将消息追加到会话存储文件。助手消息采用异步写入避免阻塞生成器，用户消息和压缩边界消息采用同步写入确保顺序一致性。会话存储支持增量写入和原子刷新，通过环形缓冲区管理内存使用，避免长时间运行会话的内存泄漏。

Sources: [QueryEngine.ts](src/QueryEngine.ts#L600-L900), [query.ts](src/query.ts#L600-L900)

### 上下文压缩策略

查询引擎实现多层次的上下文压缩策略，在保持语义完整性的同时最大化 token 效率。**历史裁剪**通过 `snipCompactIfNeeded` 实现，基于启发式规则识别可以安全删除的旧消息片段，生成压缩边界消息标记裁剪点。**微压缩**通过 `microcompactMessages` 实现，对工具调用序列生成紧凑表示，使用摘要替换详细输出，特别适用于重复的文件操作或搜索任务。

**自动压缩**通过 `autoCompactIfNeeded` 实现，当 token 计数接近上下文窗口限制时触发，调用独立的压缩代理生成对话摘要。自动压缩跟踪连续失败次数，实现熔断机制防止无限重试。压缩后的消息通过 `buildPostCompactMessages` 构建后压缩消息列表，包括摘要消息、附件消息和钩子结果消息。

**上下文折叠**通过 `applyCollapsesIfNeeded` 实现，将相关的消息分组折叠为语义单元，在需要时展开查看详情。折叠操作延迟到实际需要时执行，避免过早丢失细节。当发生提示过长错误时，折叠机制可以排空已暂存的折叠，快速释放 token 空间。

**响应式压缩**是对传统自动压缩的增强，在 API 返回提示过长错误后触发，通过智能摘要减少 token 消耗后自动重试。响应式压缩可以处理媒体大小错误，通过剥离大型附件后重试，避免因单个大文件导致整个请求失败。

Sources: [query.ts](src/query.ts#L400-L600)

## 错误处理与恢复机制

### API 错误处理策略

查询引擎对 API 错误采用分级处理策略，区分可恢复错误和不可恢复错误。**提示过长错误**（413）通过压缩和重试机制恢复，首先尝试上下文折叠排空，然后尝试响应式压缩，如果仍然失败则返回错误并终止查询。**输出 token 超限错误**通过升级输出 token 限制重试，如果仍然超限则注入恢复消息引导模型分步完成任务。

**媒体大小错误**通过响应式压缩的剥离重试机制处理，移除大型附件后重新发送请求。**模型降级错误**通过 `FallbackTriggeredError` 异常触发，切换到备用模型并清除当前助手消息，重新发送请求。**流式中止错误**通过 `StreamingToolExecutor` 的剩余结果获取机制处理，为已排队和进行中的工具生成合成的工具结果块，确保工具调用的完整性。

错误消息在某些情况下会被暂时扣留，直到确定恢复是否成功。例如，提示过长错误在响应式压缩尝试期间被扣留，如果压缩成功则错误消息永不产出，如果压缩失败则错误消息才被产出。这种扣留机制避免了中间错误状态泄漏给 SDK 调用者，特别是对于在收到任何错误后立即终止会话的客户端。

### 中止与清理机制

查询引擎通过 `AbortController` 实现优雅的中止机制，用户可以通过键盘中断或 SDK 调用中止查询。中止信号被传递到所有异步操作，包括 API 调用、工具执行和钩子执行，确保及时响应中止请求。

中止时，流式工具执行器通过 `getRemainingResults` 方法消费剩余的工具执行结果，为中止的工具生成合成的工具结果块。如果使用批量工具执行模式，则通过 `yieldMissingToolResultBlocks` 生成缺失的工具结果块。这确保了助手消息中的工具调用都有对应的用户消息工具结果，维护了消息配对的完整性。

清理机制包括芝加哥 MCP 锁释放、会话存储刷新和文件历史快照。中止时释放计算机使用锁，恢复界面可见性，避免锁泄漏导致的后续操作阻塞。会话存储在关键点刷新，确保内存中的消息被持久化到磁盘。文件历史在用户消息被接受后创建快照，支持后续的文件版本比较和回滚。

Sources: [query.ts](src/query.ts#L800-L1300), [QueryEngine.ts](src/QueryEngine.ts#L800-L1100)

## 性能优化与监控

### 流式工具执行

流式工具执行是查询引擎的关键性能优化，通过 `StreamingToolExecutor` 类实现。该类在助手消息流式到达时立即开始工具执行，与 API 响应并行运行，显著降低端到端延迟。

执行器维护一个工具调用队列，每个工具调用被封装为 `QueuedToolCall` 对象，包含工具块、助手消息、执行状态和结果消息。当 `addTool` 方法被调用时，工具调用被添加到队列并立即开始执行。`getCompletedResults` 方法返回已完成执行的调用结果，支持生成器的即时产出。

执行器处理权限检查、参数验证和错误处理。权限检查通过包装的 `canUseTool` 函数实现，参数验证通过工具的 `validateInput` 方法实现。执行错误被捕获并转换为错误消息，通过 `yield` 产出给上层调用者。

当发生流式中止或模型降级时，执行器通过 `discard` 方法丢弃待处理的工具调用，通过 `getRemainingResults` 方法消费剩余的结果，确保资源被正确释放和消息配对被维护。

Sources: [query.ts](src/query.ts#L600-L800)

### 性能监控与诊断

查询引擎通过多个检查点和度量点实现性能监控。`queryCheckpoint` 函数在关键执行点记录时间戳，用于构建查询性能剖析。`headlessProfilerCheckpoint` 函数在无头模式下记录延迟关键路径上的事件时间。

API 调用监控通过 `captureAPIRequest` 和 `logError` 函数实现，记录请求参数、响应时间和错误信息。使用量统计通过 `accumulateUsage` 和 `updateUsage` 函数累计，包括输入 token、输出 token、缓存读取 token、缓存创建 token 等。

会话监控通过 `currentLimits` 和 `extractQuotaStatusFromHeaders` 函数实现，跟踪配额使用情况和限制状态。成本跟踪通过 `calculateUSDCost` 和 `addToTotalSessionCost` 函数实现，实时计算会话成本并在达到预算限制时终止查询。

诊断日志通过 `logForDebugging` 和 `logForDiagnosticsNoPII` 函数实现，记录详细的执行信息和状态转换，用于问题排查和性能调优。错误日志通过 `logAntError` 函数实现，在蚂蚁模式下大声记录错误，帮助开发者快速定位问题。

Sources: [QueryEngine.ts](src/QueryEngine.ts#L600-L900), [query.ts](src/query.ts#L200-L400)

## 架构设计原则

查询引擎的设计遵循多项工程原则，确保代码的可维护性、可测试性和可扩展性。**单一职责原则**通过分层架构实现，每层专注于特定职责：QueryEngine 管理会话状态，query 函数处理执行逻辑，依赖模块提供核心能力。**依赖倒置原则**通过依赖注入实现，核心逻辑依赖抽象接口而非具体实现，使测试可以注入 mock 对象。

**开闭原则**通过钩子机制实现，扩展功能通过注册钩子注入，无需修改核心代码。**接口隔离原则**通过细粒度的类型定义实现，每个组件只暴露必要的接口，避免庞大的接口污染使用者。**生成器模式**通过 `yield` 和 `async function*` 实现，提供非阻塞的流式处理能力，同时保持代码的线性可读性。

**状态机模式**通过可变状态对象和 continue/return 语句实现，将复杂的流程建模为状态转换，避免深层嵌套和复杂的控制流。**策略模式**通过不同的压缩算法实现，根据上下文和配置选择合适的压缩策略。**模板方法模式**通过查询循环骨架实现，定义执行流程的框架，具体步骤由依赖模块提供。

这些设计原则使得查询引擎能够在保持核心逻辑简洁的同时，支持丰富的功能和灵活的扩展，为 Claude Code 的稳定运行和持续演进提供了坚实的基础。
