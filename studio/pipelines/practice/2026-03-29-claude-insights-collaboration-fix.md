一个月，111 次对话，499 小时。其中 79 次 AI 跑偏了方向，55 次 AI 理解错了意图。

也就是说，每 10 次对话就有 7 次需要纠正方向，每 2 次对话就有 1 次 AI 从一开始就理解错了你要什么。这不是"AI 偶尔犯错"，这是**系统性的协作失败**。

这组数据来自 Claude Code 内置的 `/insights` 命令生成的使用报告。拿到这份报告之前，我以为问题出在 AI 不够聪明。拿到之后才发现：**问题不在 AI 的能力，在人和 AI 说话的方式不匹配。**

这篇文章记录了从发现问题到修好它的完整过程，包括踩过的坑。如果你也用 Claude Code 或类似的 AI 编程助手，每天花大量时间纠正 AI 的方向，这篇可能帮你省下不少力气。

![概念图：协作断裂诊断](2026-03-29-claude-insights-collaboration-fix/material/pngs/01_协作断裂诊断.png)


## 症状：AI 很忙，但没做对事

场景一：让 Claude Code 帮忙分析企百科的用户问答质量。说了一句"帮我看看回答质量怎么样"。Claude 立刻开始设计一套完整的评估框架——先建标签体系、再设计标注规范、然后跑全量标注、最后输出多维度交叉分析报告。方案很专业，但执行下来至少要两天。

实际上我当时只需要一件事：**先抽 100 条看看，大概感觉一下问题出在哪。** 是回答质量差？还是根本没回答？是所有类型都差？还是某一类特别差？有了这个方向感，后面的全量分析才知道该分析什么。

AI 的本能是"做完整"，但很多时候完整的方案反而是浪费——因为方向还没确认就投入了全部资源。

场景二：数字员工项目，140 张数据表的配置自动化。说了一句"帮我把配置流程理一下"。Claude 直接产出了 10 个 MCP 工具的完整 API 设计、4 种数据源分支的全覆盖方案、DDL 转换规则——一套能直接交给开发的技术规格书。

但我那个时刻想要的是：**先搞清楚一张表的完整配置流程到底有几步，每步有没有坑，手动走一遍再说。** 从"手动走通一张表"到"设计全自动化方案"之间，还差着一个"验证理解是否正确"的步骤。AI 跳过了这一步。

这两个场景的共同点是：**AI 默认用最完整的方案响应，但实际工作中，最快验证方向的方案往往是最有价值的。**


## 诊断工具：Claude Code 的 /insights 命令

在说诊断结果之前，先介绍一下这个工具，因为它是整个故事的起点。

Claude Code 内置了一个 `/insights` 命令。在终端里输入这个命令，它会扫描你过去所有的对话记录，生成一份 HTML 格式的使用报告。报告长这样：

![Insights 报告截图](2026-03-29-claude-insights-collaboration-fix/screenshots/insights-report-overview.png)

报告里有什么：
- **你在哪些项目上花了最多时间**（按会话数排序）
- **你和 AI 的交互风格画像**（比如"你是一个快速迭代的项目经理，边做边改方向"）
- **摩擦点统计**（多少次 AI 理解错了、多少次方向跑偏、多少次撞上下文限制）
- **具体的改进建议**（要加什么配置、用什么功能、改什么习惯）
- **可以直接复制粘贴的配置代码**（CLAUDE.md 规则、Hook 配置、Skill 模板）

简单说就是：**让 AI 用数据复盘自己和你的配合质量，然后给出可执行的修复方案。**

跑完之后，几个数据跳了出来：

**79 次"方向跑偏"（wrong approach）**：Claude 选了不对的策略——比如过度工程化、在错误方向上磨洋工、没备份就改文件。

**55 次"意图理解错"（misunderstood request）**：Claude 从一开始就理解错了我想要什么——比如把"vibe code demo"理解成准备 Skill 定义文件，而不是编写对话式开发脚本。

**模式识别**：报告把我的交互风格描述为"给开放性指令，Claude 理解错，你纠正，Claude 再试"。一句话概括了我的日常。

![概念图：踩坑路径](2026-03-29-claude-insights-collaboration-fix/material/pngs/02_踩坑路径.png)


## 诊断：AI 的过度工程化倾向

拿到 Insights 报告后，第一反应是找规律：这 79 次跑偏，到底偏在哪？

回看记录后发现一个清晰的模式：**AI 有系统性的"过度工程化"倾向。** 你说一句模糊的需求，它默认用最完整、最复杂的方案来响应。这在某些场景下是优点（比如写一份技术规格书），但在大多数日常工作场景下是浪费——因为你还没确认方向对不对，它就已经把整套方案做完了。

另一个发现是**抽象层级不匹配**。PM 说话习惯是意图导向的——"帮我看看回答质量"，这句话的重心在"看看"（快速了解），不在"回答质量"（建一套评估体系）。但 AI 把重心放在了后者，因为"回答质量分析"在技术上是一个可拆解的明确任务。

这不是 AI 的 bug，是 LLM 的训练目标决定的：**模型被训练为"尽可能完整地回答问题"，而不是"用最小成本验证方向"。** 在编程场景下这通常没问题（你让它写一个函数，它写完整是对的），但在产品/业务场景下，"完整"往往不是第一优先级，"快速验证"才是。

还有一个有意思的发现：Insights 报告显示我"感觉"AI 帮了很大忙，但数据说 71% 的对话需要纠正方向。这个感知偏差和 METR 的研究完全吻合——他们发现经验丰富的工程师用 AI 实际慢了 19%，但自己感觉快了 20%。**如果不跑数据，你根本意识不到效率在流失。**


## 解法：三层修复

发现问题之后，修复分三层：改规则、加自动化、调行为。

### 第一层：CLAUDE.md 加沟通规则

（CLAUDE.md 是 Claude Code 的配置文件，相当于给 AI 写一份"工作手册"，告诉它你是谁、怎么干活、有什么规矩。）

在 CLAUDE.md 里加了一个新段落叫"沟通规则"，5 条：

```markdown
## 沟通规则

- **先复述再动手**: 收到复杂/模糊任务时，用一句话复述你理解的目标，
  等用户确认再开始。用户说"不对"或纠正方向时，停下来重新读完整请求，
  不要猜
- **术语必须配大白话**: 用户没有开发背景。说 Flask/Railway/CNAME 这类
  词时必须解释清楚是什么意思
- **不要给3个方案让用户选**: 直接做最好的那个，做完让用户改
- **不要在"讨论该怎么做"上花超过1轮对话**: 第2轮必须有产出
- **上下文快满时主动存档**: 检测到75%上下文使用时，把完成的任务、
  待办、关键决策存到 checkpoint 文件再继续
```

这 5 条规则每一条都针对报告里识别出的具体摩擦模式：
- "先复述再动手" → 解决 55 次意图理解错
- "术语配大白话" → 解决我不懂却不敢问的问题
- "不给3个方案" → 解决讨论过多行动太少
- "第2轮必须有产出" → 解决空转
- "主动存档" → 解决上下文超限丢进度

### 第二层：Pre-Edit Hook 自动备份

（Hook 是 Claude Code 的自动触发机制。你可以设定"每次 AI 要改文件之前，先自动执行一条命令"。）

报告指出 Claude 有覆盖文件不备份的问题。解法是在 settings.json 里加一个 Hook：每次 AI 准备编辑或写入文件时，自动先把原文件复制一份备份。

```json
{
  "matcher": "Edit|Write",
  "hooks": [{
    "type": "command",
    "command": "/path/to/pre-edit-backup.sh"
  }]
}
```

备份脚本 30 行，逻辑很简单：检查文件是否存在 → 如果是 .md/.py/.html/.svg 等常见格式 → 自动 `cp 原文件 原文件.bak`。

这样即使 AI 改坏了文件，随时能恢复。完全自动，不需要手动操作。

### 第三层：调自己的行为

最难改的其实是自己。

报告让我意识到一个模式：**我的直觉很准，但表达有延迟。** 我经常先感觉到"不对"，但需要好几轮对话才能说清楚到底哪里不对。这中间 AI 就在错误的方向上越跑越远。

对治办法：**不懂就立刻说，感觉不对就立刻打断。** 不要等想清楚了再说——"不对，停一下"比委婉地说"有些问题"效率高很多。

另一个发现是：**先说要什么东西，再说背景。** 不要说"我们之前讨论的自动投递的产品设计怎么样了"（AI 要花 20 分钟去翻历史），直接说"帮我写简历"（AI 2 分钟就开始写）。

## 验证：修复前 vs 修复后

加完规则之后变化立竿见影。

**修复前**（企百科分析 v1）：我说"帮我看看回答质量" → Claude 花 30 分钟设计完整评估框架 → 我说"太重了，先抽样看看" → 推翻重来。一个小时过去，还在对齐方向。

**修复后**（同类任务）：Claude 先复述"你的意思是先抽 100 条快速看方向，确认了再全量跑？" → 我说对 → 10 分钟出初步结论 → 确认方向后再投入全量分析。最终那个项目从全量 2588 条数据里发现了"误拒才是核心瓶颈"这个关键洞察。

**关键区别不是 AI 变聪明了，是规则让双方的工作节奏对齐了。**

- "先复述再动手"让方向错误在第 1 轮就暴露，不再第 5 轮才发现
- "不给3个方案"砍掉了无效讨论——以前 AI 总列"方案A/B/C让你选"，现在直接做最优解，不对再改
- "第2轮必须有产出"杜绝了空转——不允许连续两轮都在"讨论该怎么做"


## 行业对标：不只是我一个人踩坑

这个"人和 AI 协作低效"的问题，不是个例。

METR（一家专门测量 AI 能力的研究机构）做了一项实验：让 16 名有 5 年以上经验的开发者用 AI 完成 246 个任务。结果是：**开发者预测 AI 会让自己快 24%，实际却慢了 19%。** 更有意思的是，即使变慢了，开发者仍然"感觉"自己快了 20%——存在 39 个百分点的感知偏差。

这正好解释了为什么需要 Insights 报告：**你以为 AI 帮了你，其实在某些场景它在拖后腿，但你感觉不到。** 只有系统性地回顾数据，才能发现这种盲区。

Anthropic 官方文档也给了一个关键数据：**CLAUDE.md 大约有 150-200 条指令的有效上限**，超过这个数量遵从度就会明显下降。而且 CLAUDE.md 是"建议性的"（约 80% 遵从率），如果某件事必须 100% 发生，应该用 Hook 而不是写在 CLAUDE.md 里。这解释了为什么我们的修复方案是"规则 + Hook"两层而不是只写规则。

Arize AI 团队做了一个更系统化的实验：用 LLM Eval 的反馈信号（不是简单的"对/错"）来迭代优化 CLAUDE.md，在 SWE-Bench 上把测试通过率提升了 5.19%。这验证了"从协作摩擦中提取结构化经验，反哺配置文件"这条路是走得通的。

Google Chrome 团队的 Addy Osmani 分享了他 2026 年的 AI 编码工作流，核心是三件事：写详细 spec 再开始、一次只做一件事、建立反馈循环（AI 写 → 工具检查 → AI 修 → 人审 → 沉淀经验）。最后一条就是 Insights 报告的思路——跨 session 的长期反馈循环。

但要注意适用边界：Hacker News 上一个从 Cursor 切换到 Claude Code 的用户指出，**纯终端 AI 在前端视觉任务中反而更慢**——"用文字描述视觉变化，再在脑中映射回像素，比自己动手还慢。" 这说明 Insights 报告不应该一刀切地优化所有任务，而应该按任务类型标注"AI 在哪些场景真的提效"。


## 可迁移的方法论

从这个具体经验里，可以抽象出 3 条通用原则：

### 原则 1：AI 协作的瓶颈不在 AI 的能力，在人机接口的设计

大多数人优化 AI 协作的思路是：换更强的模型、写更好的提示词、给更多上下文。但实际瓶颈往往是更基础的东西——双方对"要做什么"的理解是否对齐。

**如果你遇到类似情况**：先别急着改提示词。跑一次使用报告（Claude Code 用 `/insights`，Cursor 看 Settings > AI Logs），看看摩擦到底发生在哪个环节。

### 原则 2：把经验固化成规则，比每次手动纠正高效一百倍

发现一个问题 → 手动纠正 → 下次还犯。这是 111 次对话中反复出现的模式。

修好它的方式不是"记住下次要纠正"，而是**写成规则让系统自动执行**。CLAUDE.md 就是这个"规则文件"——AI 每次开始工作前都会读它，相当于你提前把所有的纠正都说完了。

**如果你遇到类似情况**：每次纠正 AI 之后，问自己"这个纠正能不能变成一条规则？"如果能，写进配置文件。

### 原则 3：让 AI 复盘自己的表现，比人类自己复盘更客观

人类复盘总是带有偏见——记住的是印象最深的几次，不是全部。AI 分析 111 次对话的数据，发现的模式是我自己绝对看不出来的（比如"每10次对话有7次跑偏"这个比例，我体感最多3成）。

**如果你遇到类似情况**：定期让 AI 分析自己和你的协作数据。不是让它"反思"（那只会产出空洞的自我批评），而是让它跑数据——多少次成功、多少次失败、失败的模式是什么。

## 本期关键词

**Insights Report（使用洞察报告）** -- Claude Code 内置的 `/insights` 命令生成的分析报告，基于你的历史对话数据，统计成功率、摩擦点、交互模式。相当于让 AI 做一次全面的"协作体检"。其他工具（Cursor、Windsurf）暂时没有同等功能，但思路可以借鉴。

**CLAUDE.md** -- Claude Code 的项目级配置文件，放在项目根目录。AI 每次开始工作前会读这个文件，按里面的规则行动。相当于给 AI 写一份"入职须知"：你是谁、怎么干活、有什么红线。是把"每次手动纠正"变成"一次性规则"的核心机制。

**Hook（钩子）** -- Claude Code 的自动触发机制。你可以设定"在某个事件发生时，自动执行一条命令"。比如"每次AI要改文件前，自动备份原文件"。技术上是在 settings.json 的 hooks 字段里配置，支持 PreToolUse / PostToolUse / PreCompact / Stop 等时机。

**抽象层级不匹配（Abstraction Level Mismatch）** -- 人机协作中最常见也最隐蔽的摩擦源。人说的是意图层（"让面试官看懂我做了什么"），AI 理解的是执行层（"Flask部署到Railway"）。解法不是让人学技术语言，而是让 AI 先复述理解、再动手执行。

**Wrong Approach（方向跑偏）** -- Insights 报告中的一个摩擦分类。指 AI 选择了不恰当的执行策略——比如过度工程化、在错误方向上反复微调、没有备份就修改文件。和"意图理解错"是两回事：理解对了但执行策略不对。
