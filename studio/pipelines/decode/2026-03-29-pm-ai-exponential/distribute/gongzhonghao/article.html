地基在动 -- 当 AI 能力指数增长时，产品经理该怎么活

原文: Cat Wu (Anthropic Claude Code 产品负责人) | 拆解: Deep Decode

41 倍。16 个月。

METR 的测试显示，Opus 4.6 能完成需要人类大约 12 小时的软件任务，而 16 个月前的 Sonnet 3.5 只能处理 21 分钟级别的任务。这个数字的含义不是「AI 更快了」，而是：花三个月写的路线图，等写完模型可能已经换了两代。精心设计的 workaround，下个模型可能原生就支持了。

Anthropic Claude Code 产品负责人 Cat Wu 发了篇博客，聊她们团队的产品经理现在到底怎么干活的。一句话总结：传统 PM 那套玩法，在模型能力指数级增长的时代，已经不够用了。这倒也不是什么惊人之语。但 Anthropic 自己的 PM 负责人拿出具体案例来说，就值得更认真地拆一拆了。

地基在动

Cat Wu 用了一个比喻：你在一块正在上升的地面上盖房子。

传统产品管理有个隐含假设：项目开始时技术能做什么，结束时大体也差不多。路线图定好，团队按节奏推进，一切有条不紊。但 AI 模型的进化速度，把这个假设给掀了。

她自己有个保留测试项目：让 Claude Code 给 Excalidraw 加一个表格工具。2024 年 10 月用 Sonnet 3.5 试，基本做不出来。2025 年 6 月用 Opus 4 试，偶尔能成功，可以提前录好 demo 用。到了现在的 Opus 4.6，已经可以当场现场演示了。

同一个测试项目，三个版本，完全不同的结果。

关于 METR 那个 41 倍的数据，值得多说两句。MIT Technology Review 今年 2 月发了一篇文章，标题直接叫「This is the most misunderstood graph in AI」。他们指出，METR 的纵轴测量的是「AI 在 50% 成功率下能完成的人类任务时长」，不是 AI 自主运行的时间。UC Berkeley 的 Inioluwa Raji 进一步质疑：人类完成时间是否真的能衡量任务难度？而 METR 自己 2026 年 1 月更新的 Time Horizon 1.1 数据显示，Opus 4.6 实际对应的人类任务时长是 14.5 小时，比 Cat Wu 引用的 12 小时还高。

这个数据在趋势上是可靠的，但在精度上值得打折。关键不是 41 倍还是 69 倍，而是增速是指数级的，且在加速 -- METR 自己的数据表明，能力翻倍周期已经从 7 个月缩短到约 4 个月。在这种速度下，任何超过一个季度的计划都有可能在执行期间被底层能力的跃迁掀翻。

Side Quest 文化：PRD 的替代品

那怎么办呢？Anthropic Claude Code 团队的做法是：把长期路线图换成了短冲刺，外加一种叫「Side Quest」的机制。

Side Quest 就是团队里任何人 -- 工程师、设计师、PM 都行 -- 发起的自主实验。不在正式路线图上，可能就是某天下午冒出来的一个念头。Cat Wu 的原话是：「Wrong bets are cheap」 -- 当原型只需要一个下午，赌错了代价几乎为零。

Claude Code 里好几个现在大家在用的功能，就是这么来的：Claude Code on Desktop、AskUserQuestion 工具、Todo lists。这些功能的共同点是：没有人写过 PRD，没有经过传统的评审流程。有人觉得这个想法可能行，下午就做了个原型，内部用户试了觉得好用，然后就上线了。

Pragmatic Engineer 的 Gergely Orosz 做了更深入的报道：Claude Code 团队平均每个工程师每天发布约 5 次，每个新功能会经过 10 个以上原型迭代。更有意思的是，Claude Code 本身就是 Side Quest 的产物 -- 起源于一个工程师写的 CLI 小工具，最初功能只是在终端显示他正在听什么音乐。

但这里有个 Cat Wu 刻意没展开的前提条件：Side Quest 文化能成立，是因为 Anthropic 具备三个特殊条件。第一，他们在做 AI-native 产品，模型本身就是产品，所以原型的验证周期极短。第二，团队小且资深。第三，内部 dogfooding 提供了即时信号 -- 自己做的东西自己每天在用。

Battery Ventures 在他们的分析中就指出了这个问题：企业级公司想采用这种「build-first」文化时，最大的障碍是「structured rituals they can't easily shed」-- 那些围绕 PRD、评审会、排期会建立的整套仪式，不是说换就能换的。

Side Quest 的本质不是「不要计划」，而是「计划的颗粒度要跟模型迭代速度匹配」。当模型每 4 个月翻一倍，三个月的路线图就是在用过去的能力边界来约束未来的可能性。

原型吃掉文档

这是整篇文章里最值得关注的变化。

Cat Wu 给团队的建议是：写完 spec 之后，把它发给 Claude Code，看看能不能直接把东西做出来。她举了个例子 -- 团队成员 Noah 写了一份 plugins 的 spec，直接喂给 Claude Code，出来的原型质量高到成了最终上线版本的基础。另一位成员 Conner 则手工写了一套 eval（评估测试），定义了什么情况下功能算「成功」，什么情况下算「失败」。

PM 的产出物，变了。以前的核心交付是文档：PRD、需求说明、流程图。现在的核心交付变成了两样东西：可运行的原型和可衡量的 eval。

Prompt，正在成为新的 PRD。

这不是 Anthropic 一家在这么干。微软 CPO Aparna Chennapragada 最近在 Lenny's Podcast 上说了一句话，跟 Cat Wu 的意思完全呼应：「Prompt sets are the new PRDs... demos before memos.」她要求团队推进新项目时，不光要写文档，还要拿出原型和对应的提示词集合。她的定位是：PM 正在从「需求定义者」变成「品味把关者」（tastemaker） -- 当 AI 能快速生成 100 种方案时，判断哪个方案好才是核心能力。

Figma CEO Dylan Field 从设计工具的视角给了另一个佐证。Figma 2025 年的用户调研显示，72% 的受访者认为 AI 工具推动了角色扩展，56% 的非设计师现在「大量参与」设计工作（去年这个数字是 44%）。Dylan Field 的判断是：「We're all product builders, just with different specializations.」

Aakash Gupta 在分析 Anthropic 的产品流程时更直接：「Anthropic skips the spec step altogether -- no 10-page PRD, no detailed requirements doc... they just build a working prototype.」

「原型吃掉文档」这件事的影响，远比它表面看起来的深。它不只是改变了 PM 的产出物，它改变了 PM 的能力模型。以前 PM 的核心能力是「写得清楚」-- 把模糊的需求翻译成工程师能执行的文档。现在这个能力的价值在快速下降，因为 AI 可以直接从模糊的描述生成原型。取而代之的核心能力是「看得准」-- 在 100 个原型里挑出最对的那 2-3 个。

模型升级即产品升级

Cat Wu 提到的第三个变化，才是最容易被忽略的。

每次新模型发布，团队会重新翻一遍之前「做不到」的功能清单，拿新模型再试一次。同时，还会清理掉旧模型时代堆上去的 workaround。

她举了个 Todo list 的例子。早期模型不会主动把完成的任务勾掉，团队就在 system prompt 里加了提醒。后来新模型出了之后，这个行为变成了自带的，提醒也就删了。整个 Opus 4.6 的 system prompt 和工具描述，比上一代精简了 20%。

这背后的逻辑是：今天的 hack，就是明天的技术债。但在 AI 产品领域，这个「明天」来得特别快，可能就是两三个月后的事。

Chrome 集成的案例更有意思。团队观察到用户在用 Claude Code 写网页应用时，会手动切到浏览器去测试，然后再手动把报错信息复制回来。Cat Wu 的判断是：如果用户在自己搭这种桥，那说明这个桥应该内置到产品里。

这个观察方法本身就是一个值得记住的产品 insight -- 看用户在哪里手动做「胶水工作」（glue work），那里就是产品机会。

做简单的事

Anthropic 内部有个指导原则，Cat Wu 反复提到：Do the simple thing.

在 Agent 系统中，复杂度是指数级放大失败的。多加一层 workaround，失败模式就多一类。所以 Cat Wu 的建议是：先用最笨的方法做。如果模型今天不够好，先别急着搭复杂的脚手架。等下一代模型出来，很可能这个问题就不存在了。

这个原则，说起来容易，做起来挺反直觉的。工程师的本能是遇到问题就解决问题。但在 AI 产品的语境下，有些问题最好的解法是......等等看。

Pragmatic Engineer 的报道佐证了这一点：Claude Code 的技术栈选择（TypeScript、React、Ink、Bun）被刻意选为「on distribution」-- 也就是模型训练数据中出现频率最高的技术栈。不是因为这些技术最好，而是因为让 AI 在它最熟悉的领域工作，比硬用一个更优雅但更小众的方案，效果好十倍。

「建造鸿沟」-- 从实现成本到方向选择

如果把视野拉远一点，Cat Wu 描述的这些变化指向一个更大的趋势，可以称之为「建造鸿沟」。

以前，从想法到产品之间隔着一条宽宽的河：需要工程师、设计师、测试、排期、开发、联调。这条河就是实现成本。PM 的工作本质上是决定哪些想法值得付出过河的代价。

现在 Claude Code 这类工具把河水抽干了。从想法到原型，一个下午就到。

河没了，但目的地变得更多了。快速抵达一百个方向成为可能，但只有两三个是对的。判断哪几个方向值得走，这才是新时代 PM 最核心的能力。

Battery Ventures 的分析文章验证了这个判断：「AI 时代产品团队面临的核心张力是：执行速度在加快，但方向选择在变难。」以前的瓶颈是工程产能，PM 的核心价值是排优先级、分配资源。现在的瓶颈变成了：到底该建什么？

提效的悖论

LogRocket 2026 年的分析中提到了一个有意思的现象：很多团队发现，产品团队出原型更快了，研发团队写代码也更快了，但整个产品线的交付效率......几乎没变。

为什么？因为产品团队能更快地做出东西，但花了更多时间在争论该做哪个东西。选项变多了，决策反而变慢了。

这就是「建造鸿沟」的另一面：实现的鸿沟在缩小，但「该做什么」的鸿沟在变大。

PM 的 spec 几乎成了「易腐品」。六周前写的需求，到了今天可能已经有完全不同的解法了。

盲区：这篇文章没说的

第一，幸存者偏差。Anthropic 是全球最前沿的 AI 公司之一，他们的团队结构、人才密度、技术能力都不是普通公司能复制的。「Side Quest 下午就做出一个功能」这种事，在一个 50 人的中型 SaaS 公司，可能完全走不通。

第二，METR 图的方法论争议。她引用了 41 倍的数据但没有提及 MIT Technology Review 和学术界对这张图的批评 -- 这张图「在社交媒体上被频繁传播，但常常缺少上下文」。人类任务时长是否真的能衡量 AI 能力，这是一个尚未定论的方法论问题。

第三，中大型组织的适用性。文中引用的 Decagon 和 Datadog 的例子，都是偏技术导向的公司。对于那些有复杂合规要求、多层审批流程、跨部门依赖的传统企业，这套「原型优先、文档靠后」的做法需要大幅改造才能落地。Aha.io 的分析就指出：「Although the rate of change feels profound, the timeline for true transformation will be longer than the current hype suggests.」

第四，PM 真的不会被替代吗？Cat Wu 的叙事是「PM 进化」，但 Product School 的数据显示，美国「管理类职位」在 2025-2034 年的增长预期高于平均水平 -- 这说明市场目前还不认为 PM 会被 AI 替代。但这个数据是在 Agent 能力还相对初级时做的预测，如果 METR 的增速继续保持，两年后的情况可能完全不同。

对 AI 从业者/实践者来说意味着什么

从 Cat Wu 这篇文章中，可以提炼出三条直接可用的判断：

1. 计划颗粒度 = 模型迭代周期。不要做超过一个季度的详细计划。定方向可以看一年，但具体功能规划最多看 6-8 周。每次模型升级都是重新评估「能做什么」的时机。

2. 看用户在哪里做胶水工作。Cat Wu 发现 Chrome 集成需求的方法 -- 观察用户在哪里手动搭桥 -- 这是一个通用且低成本的产品 insight 方法。

3. 原型 > 文档，但 eval > 原型。做出来不难，难的是知道做得对不对。写 eval、定义成功标准、设计验证方法 -- 这些才是 PM 应该花最多时间的地方。

本期关键词

Dogfooding -- 字面意思"吃自己的狗粮"。指团队日常使用自己开发的产品，从而获得第一手用户反馈。Anthropic 团队每天用 Claude Code 写 Claude Code，这就是典型的 dogfooding。对标国内说的"内部灰度"但更激进 -- 不是灰度测试，是全量日常使用。

Side Quest -- 借用游戏术语"支线任务"。在 Anthropic 的语境下指任何人可以自主发起的、不在正式路线图上的小型实验。核心逻辑：原型成本极低时，失败的代价趋近于零，所以鼓励大量尝试。Claude Code 本身就起源于一个工程师的 Side Quest。

Eval -- Evaluation 的缩写，AI 领域的"考试"。不是传统的 QA 测试，而是专门为 AI 行为设计的评估体系：定义什么算"做对了"、什么算"做错了"，通常由 prompt + 判定规则 + 测试用例组成。Cat Wu 认为 eval 正在取代 PRD 成为 PM 最核心的产出物。

On Distribution -- 机器学习术语，指"在训练数据分布内"。Claude Code 选择 TypeScript/React/Bun 技术栈，不是因为这些最好，而是因为它们在模型训练数据中出现频率最高，AI 对它们最"熟悉"。反义词是 out-of-distribution（OOD），意味着模型容易犯错。

Tastemaker -- 品味把关者。微软 CPO Aparna Chennapragada 用这个词定义新时代 PM 的核心角色：当 AI 能快速生成 100 种方案时，PM 的核心价值不再是定义需求，而是判断哪个方案最好。从"写得清楚"到"看得准"的能力跃迁。

Glue Work -- 胶水工作。指用户在两个工具/流程之间手动做的桥接操作。Cat Wu 观察到用户在 Claude Code 和 Chrome 之间手动复制粘贴，这种 glue work 就是产品机会的信号。看到 glue work 就应该想"这个桥能不能内置"。

Time Horizon -- METR 提出的 AI 能力度量指标。衡量的是"AI 在 50% 成功率下能独立完成的、相当于人类多长工作时间的任务"。Opus 4.6 的 time horizon 是 ~14.5 小时，意味着它能可靠完成需要人类大约两个工作日才能完成的软件任务。
