# Phase 1 结构解剖 -- Prefill-as-a-Service

## 1. 核心论点
Prefill-Decode 分离过去只能在单数据中心的 RDMA 胖管道内部做。Moonshot+清华通过 hybrid attention 大幅压缩 KVCache，再叠加三个系统级机制（选择性卸载 / 双时间尺度调度 / 混合前缀池），把 prefill 变成跨数据中心的"独立服务"（PrfaaS），用商用百 Gbps 以太网就跑得通。1T 模型实测 54% 吞吐提升。

## 2. 隐含假设
1. 算力地理分布已经显著不均衡（一些 DC 有 H200 多，一些只有 H20）
2. 长请求是长尾但占了 prefill 大头成本，值得单独对待
3. KVCache 作为一等公民（Mooncake 提出）从 intra-cluster 被推到 cross-DC
4. 架构（hybrid attention）和系统（调度）是共生关系，不是替代
5. 100 Gbps 级别的商用 VPC peering 已经是云厂商标配

## 3. 关键证据（最强 3 条）
- Table 3: MiniMax-M2.5 @ 32K 需要 59.93 Gbps，Ring-2.5-1T 只需 2.59 Gbps（23× 差距）
- Table 6: 1T 模型上 32 H200 + 64 H20 vs 96 H20 同构基线 → 54% 吞吐 + TTFT 砍半
- 实际带宽占用仅 ~13 Gbps（100 Gbps 链路的 13%），证明商用以太网够用

## 4. 盲区
1. 论文中"跨数据中心"实际是同地域相邻集群，真跨大洲（中美/中欧）未验证
2. SLO 定在 40 tokens/sec 属于中等松弛，严苛 streaming 场景未测
3. 所有实验基于 Kimi 自家 1T 模型，迁移到外部模型的可复现性未知
4. 跨 VPC 的 KVCache 数据安全/隔离完全没提
5. 商用带宽成本（$/月 TCO）缺失，只证明了技术可行
6. 单次 prefill 卸载带来的 2.22s mean TTFT 对秒级交互 UI 依然偏高
