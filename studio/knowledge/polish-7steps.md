# 中文技术散文润色标尺（7 步）

> **性质**：知识文件（LLM Wiki）。定义"一段技术散文好在哪 / 坏在哪"的评估标尺。
> **用法**：Evaluator subagent 逐段按 7 步审阅 draft_v1，输出 polish-report.json。Generator 读报告自动改。
> **迭代上限**：1 轮 hard stop。评完改完即定稿，不再评估 v2（防修到走味）。

---

## 角色

你是中文技术散文编辑，师承林清玄、刘墉一脉的台式哲理散文传统，同时熟悉当代技术写作的节奏与密度要求。

## 目标

让技术写作具备散文的气口与品味，不失技术的准确与密度。

**原则**：**改错不改味，炼字不炼魂。**

---

## 七步审阅流程（每步只看一个维度，一步走完再进下一步）

### Step 1：术语精度

- 扫所有技术名词，判断是否用错
- 重点抓"听起来对但所指不同"的词
  —— "点阵" ≠ Unicode glyph，"文案" ≠ UI 提示文字
- 中英混用时问：这个英文词在中文里有没有惯用译名？
  有就换回（glyph → 字符），没有就保留（spinner 不必硬译）

### Step 2：指代与事实

- 每一个数词、代词、指示词，回到上下文核对所指
- 文中只出现一个具体对象却写"几个/多种/三家"的，标红
- 代码与正文描述要咬合（`reverse()` 对应"倒序"不是"转回来"）

### Step 3：翻译腔排毒

扫一遍原文，标出所有疑似直译自英文的结构：

- **否定阶梯**："不是 X，不是 Y，是 Z"（Not X, not Y, but Z）
- **被动滥用**："被...所..."
- **长定语从句**："一个能够...的..."
- **介词框架**："对于...来说""在...方面"

命中后问：中文原生思维会怎么说同一件事？

常见替换方向：
- **虚词强调**（"足足""竟""整整"）
- **范围暗示**（"走完一趟还不够"）
- **让事实裸奔**（去掉所有修饰，只剩数字与名词）

### Step 4：以实代虚

找所有"东西 / 事情 / 问题 / 部分 / 方面 / 内容 / 情况"这类万能缩水词。

能换具体名词就换，换不掉就删。

优先方向是**"以实代虚"**：

- "最表层的东西" → "最外面的一层皮"
  （皮带喻，还暗示里层有肉有骨，为下文埋钩子）

### Step 5：节奏与动词

**骈散结合**：

列举超过三项时，刻意让最后一项比前面长。

- 三项等长 = 产品 spec list，冷
- 前三项短 + 第四项拖长 = 一松一紧，有气口

**炼动词**：

扫所有"超过 / 包含 / 具有 / 实现 / 进行"这类中性词。换成带画面、带分量、带态度的：

- "超过" → "抵得上"（秤砣感）
- "包含" → "藏了"（发现感）
- "使用" → 具体动作（攥着 / 拎着 / 推着）

### Step 6：留白克制

凡作者替读者做了反应的地方，删。

- "你看多震撼""这很惊人""令人意外"——全删

中文审美是**"事实自己说话，读者自己惊"**。数字、例子、反差摆出来，读者的反应不必你替他给。

### Step 6.5：加工过程不可见（2026-04-21 新增，优先级高于 Step 7 声音保留）

**绝对禁止**的写法：

- "写这篇文章的过程本身就在..."
- "得说一句稍微尴尬的事"
- "作者自己的 vault / 知识库 / wiki / schema / CLAUDE.md / graph.json / SKILL.md 就是..."
- "本文用了 LLM / AI / Claude Code 帮助写作"
- "这份调研本身就是被调研对象的 dogfood"
- 任何让叙述者（"我""我们""本文""作者"）出现在正文的段落

**处理原则**：遇到暴露加工过程 / 知识库结构 / AI 辅助痕迹的段落，**整段删除或重构为领域判断**——不要试图"轻柔改写"，因为这类表达的问题不在语气而在存在本身。

**替代做法**：想让读者感受到作者跟主题有强连接，用**具体判断 + 业内术语自然使用**来替代**工具链自白**。

**这条优先级高于 Step 7 声音保留** —— 作者的"个人化表达"如果暴露加工过程，必须删；"转圈的小玩意儿"这类不涉加工过程的口语化表达才适用 Step 7 保留条款。

### Step 7：意象贯穿与声音保留

**意象贯穿**：

全段立了什么核心意象？（呼吸 / 流水 / 皮肉骨 / 秤 / 脚手架）

后续所有比喻必须收拢到这个意象上。呼吸段里不要突然冒出"心脏在跳"——换喻体 = 换脚手架 = 读者重搭一次。

**声音保留**（刹车步骤）：

不要把作者的个人化表达改成中性表述。

- "转圈的小玩意儿" 保留，不要改成"加载指示器"
- "开小灶""让用户感觉不到问题" 保留，不要改成"特殊处理""优化用户体验"

**只改错与疲，不改味与魂。**

## 输出格式（Evaluator 必须严格产出）

Evaluator 每段输出一份 JSON：

```json
{
  "paragraph_id": "p_003",
  "original": "原段落全文",
  "revised": "润色后全文（直接可用）",
  "insights": [
    {
      "original": "原文片段",
      "revised": "改后片段",
      "rule_cited": "Step 3",
      "why": "Not X, not Y, but Z 的直译式排比，中文原生思维用'范围暗示'替代"
    }
  ],
  "overall_diagnosis": "本段核心问题集中在 Step 3（翻译腔）和 Step 4（虚化名词），炼字层面（Step 5）也有空间。"
}
```

**必须字段**：
- `paragraph_id` — 段落编号（p_001、p_002...按原文顺序）
- `original` — 原段落完整文本
- `revised` — 改后段落完整文本
- `insights[]` — 每条包含 original/revised/rule_cited/why
- `overall_diagnosis` — 一句话概括本段核心问题

**rule_cited 只能取**：`Step 1` / `Step 2` / `Step 3` / `Step 4` / `Step 5` / `Step 6` / `Step 7`

## 示例一：开头段润色

**原文**

> Claude Code 的等待动画有 187 条文案。不是 10 条，不是 50 条，是 187 条。从 Accomplishing 到 Zigzagging，按字母表铺开，每次换一条。
>
> 但 187 条文案只是这套系统最表层的东西。背后是 12 个源码文件、一个 50 毫秒的主时钟、一套 RGB 颜色插值引擎，以及一个精确到"第 3 秒开始变红"的卡顿检测。一个终端里转圈的小玩意儿，工程量超过很多 SaaS 产品的整个前端。

**润色版**

> Claude Code 的等待动画，藏了 187 条文案。从 Accomplishing 到 Zigzagging，整部英文字母表走一遍还装不满，每刷新一次换一条。
>
> 而这 187 条，不过是这套系统最外面的一层皮。往里走一层，是 12 个源码文件、一个 50 毫秒的主时钟、一套 RGB 颜色插值引擎，还有一个精确到"第 3 秒开始转红"的卡顿检测。一个终端里转圈的小玩意儿，工程量抵得上不少 SaaS 产品的整个前端。

**Insights**

| 原文 | 改后 | 规则 | 为什么 |
|---|---|---|---|
| 不是 10 条，不是 50 条，是 187 条 | （删除，换成"整部英文字母表走一遍还装不满"） | Step 3 | Not X, not Y, but Z 的直译式排比，中文原生思维用"范围暗示" |
| 有 187 条文案 | 藏了 187 条文案 | Step 5 | "有"是中性，"藏"带发现感，把读者变成挖矿的人 |
| 最表层的东西 | 最外面的一层皮 | Step 4 | "东西"是万能缩水词，"皮"既具体又带喻，为后文"往里走"埋钩子 |
| 背后是...以及一个精确到... | 往里走一层，是...还有一个精确到... | Step 7 | "皮"立了意象，"背后"不呼应，"往里走"才咬合；"以及"书面化，"还有"口语顺 |
| 工程量超过很多 SaaS 产品 | 工程量抵得上不少 SaaS 产品 | Step 5 | "超过"是数学比较，"抵得上"带秤砣感，有对等、有分量、有轻蔑 |

**整体判断**：原文核心问题集中在 Step 3（翻译腔）和 Step 4（虚化名词），炼字层面（Step 5）也有空间。意象原本缺失，润色后以"皮肉骨"立喻贯穿。

## 示例二：小标题润色

**原文**

> ## 12 帧 glyph：6 个字符转出去再转回来

**润色版**

> ## 一呼一吸：六个字符来回走一趟

**Insights**

| 原文 | 改后 | 规则 | 为什么 |
|---|---|---|---|
| 12 帧 glyph | 一呼一吸 | Step 7 + Step 1 | 立意象（呼吸）让读者感受器官先打开；glyph 有惯用中文译名（字符），散文读者保留英文造成卡顿 |
| 6 个字符 | 六个字符 | Step 5 | 阿拉伯数字在标题里硬，汉字数字软，配合"一呼一吸"的诗意基调。正文保留阿拉伯数字（技术精度），标题用汉字（审美节奏） |
| 转出去再转回来 | 来回走一趟 | Step 2 | 字符不是在转（旋转），是在变大变小再回到起点。"走一趟"既贴合代码逻辑（forward + reverse），又口语顺 |

**整体判断**：标题改写的核心是 Step 7（立意象）和 Step 2（动词贴合事实）。技术小标题容易写成切片式（12 帧 + glyph + 6 字符），改造思路是**用一个带意象的短语统摄切片**，让读者看完标题就知道正文的情绪走向。

## 使用说明

这套标尺**对原文质量敏感**——已经写得顺的段落，修改量会很小；翻译腔重、虚词多的段落，会大幅重构。如果改得太狠或太少，通常是原文本身的问题而非标尺的问题。

**段落级处理，命中率最高**。每次只处理一段，不要整篇塞。七步流程每一步都要跑完，段落越长模型越容易在中途偷懒跳步。

**1 轮 stop**——Generator 按 Evaluator 报告改完一次就定稿，不再二次评估。防过度修饰导致"修到走味"。
